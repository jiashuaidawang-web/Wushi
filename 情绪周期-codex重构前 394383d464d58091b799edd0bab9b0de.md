- 系统总定位-定稿
    
    下面是《顿悟股道》抽象成系统后的**理论模型定稿**。后续 PRD、页面、DDL、接口、引擎，都以这套模型为准。
    
    **一、系统总定位**
    
    系统不是行情统计工具，而是：
    
    **市场周期与主线龙头推演系统**
    
    它每天回答 5 个核心问题：
    
    1. 现在是什么“时”：市场处于哪个周期阶段？
    2. 市场有没有“机”：是否存在可参与的主线机会？
    3. 合力在哪里：资金、情绪、题材是否聚焦？
    4. 龙头是否走出：谁在竞争上位，谁只是跟风？
    5. 风险是否兑现：行情是在发酵、加速，还是退潮、收割？
    
    核心判断链路：
    
    ```
    市场周期 -> 资金供需 -> 情绪状态 -> 主线识别 -> 龙头竞争 -> 分歧一致 -> 风险兑现
    ```
    
    **二、第一原则：周期优先**
    
    系统的最高层不是个股，也不是板块，而是周期。
    
    周期决定策略边界：
    
    ```
    熊头：杀估值、杀信心，少做
    熊中：反弹多、陷阱多，只看结构机会
    熊末：恐慌释放、政策修复、妖股活跃
    修复期：游资题材先活跃，机构股建仓洗盘
    牛初：趋势龙头、券商/核心板块带市场转强
    牛中：主升浪、板块轮动、赚钱效应扩散
    牛末：龙头滞涨、补涨乱飞、风险开始聚集
    ```
    
    系统必须先判断“现在适合什么模式”，再判断“谁值得看”。
    
    **三、第二原则：个股从属于大势和主线**
    
    书里反复强调不能陷入个股思维。系统也不能以个股为中心。
    
    正确层级：
    
    ```
    大势 > 主线 > 板块 > 龙头 > 个股技术点
    ```
    
    个股只有在下面条件成立时才有系统意义：
    
    ```
    大势支持
    主线成立
    板块有持续性
    个股有地位
    资金承接健康
    风险没有兑现
    ```
    
    否则，再好看的个股也只是噪音。
    
    **四、情绪周期模型**
    
    情绪不是简单的“冰点、高潮”，而是资金与人性在周期中的表现。
    
    核心状态：
    
    ```
    恐慌冰点
    弱修复
    赚钱效应扩散
    主线发酵
    高潮一致
    高位分歧
    亏钱效应扩散
    退潮杀跌
    ```
    
    关键是判断真假：
    
    ```
    真冰点：跌够、杀透、资金开始回流、亏钱效应衰竭
    假冰点：只是下跌中继，反弹后继续杀
    真修复：主线出现、涨停质量提升、亏钱效应收敛
    假修复：一日游轮动，次日无承接
    ```
    
    所以系统不能只给情绪标签，必须给“证据链”和“置信度”。
    
    **五、主线模型**
    
    主线是资金合力的方向，不是新闻题材本身。
    
    主线生命周期：
    
    ```
    点火 -> 发酵 -> 确认 -> 加速 -> 一致 -> 分歧 -> 再一致 -> 衰退 -> 退潮
    ```
    
    主线判断指标：
    
    ```
    强度：涨停数、涨幅、成交额、资金净流入
    持续性：连续上榜、连续活跃、断板后修复能力
    扩展性：从龙头扩散到中军、补涨、低位
    竞争性：是否压过其他题材
    带动性：龙头是否能带动板块与市场情绪
    ```
    
    一句话：
    
    **主线不是第一天涨得多，而是能连续吸引资金接力。**
    
    **六、龙头模型**
    
    龙头不是预测出来的，是竞争出来的。
    
    系统应跟踪“龙头候选池”，而不是直接宣布某股是龙头。
    
    龙头类型：
    
    ```
    市场总龙：影响全市场短线情绪
    主线龙头：代表当前核心题材
    分支龙头：代表主线内部细分方向
    中军：大成交核心承接股
    补涨龙：主线后期低位扩散股
    妖股：弱势或混沌环境下的独立情绪载体
    趋势龙：机构资金推动的中长期趋势核心
    ```
    
    龙头上位证据：
    
    ```
    高度领先
    换手健康
    带动板块
    分歧后转强
    同题材竞争胜出
    弱转强或强更强
    市场情绪跟随它变化
    ```
    
    系统要表达的是：
    
    **谁在上位，谁在掉队，谁只是跟风。**
    
    **七、分歧一致模型**
    
    这是书里最重要的操作思想之一。
    
    不是“分歧就买”，而是：
    
    ```
    主线确认后的分歧转一致，才有价值
    高位一致后的再一致，往往是风险
    ```
    
    状态拆解：
    
    ```
    弱分歧：正常换手，承接良好
    强分歧：资金分歧大，但核心股抗住
    分歧转一致：换手充分后重新封强
    一致加速：缩量、快速上涨、成本断档
    一致过热：场外资金难接，风险升高
    退潮分歧：核心股破位，亏钱效应扩散
    ```
    
    系统必须识别“健康分歧”和“退潮分歧”。
    
    **八、风险模型**
    
    风险不是最后一步，而是贯穿全流程。
    
    核心风险信号：
    
    ```
    周期不支持
    主线无持续性
    龙头高位缩量加速
    成交天量后无人承接
    龙头跌破生命线
    板块强度下降
    后排大面增加
    亏钱效应扩散
    补涨乱飞
    明牌兑现
    ```
    
    书里的本质是：
    
    **行情最终都是利益兑现与财富再分配。**
    
    所以系统必须能提示：
    
    现在是机会扩散，还是进入收割阶段。
    
    **九、趋势龙模型**
    
    机构趋势龙和游资情绪龙必须分开。
    
    趋势龙判断链路：
    
    ```
    市场周期支持
    行业周期/政策支持
    板块趋势走强
    个股强于指数
    长期均线修复
    平台突破
    量价健康
    RS/RSI 强势
    ```
    
    趋势龙更重：
    
    ```
    周期
    行业
    基本面预期
    中长期资金
    周线结构
    平台突破
    波段节奏
    ```
    
    游资龙更重：
    
    ```
    题材
    情绪
    高度
    换手
    连板
    分歧一致
    短线地位
    ```
    
    二者不能混成一个“龙头评分”。
    
    **十、系统最终理论架构**
    
    最终抽象成 6 个核心引擎：
    
    ```
    1. 周期识别引擎
    判断市场大周期、情绪阶段、策略边界。
    
    2. 主线识别引擎
    判断资金合力方向、题材生命周期、主次关系。
    
    3. 龙头竞争引擎
    跟踪市场总龙、主线龙、分支龙、中军、补涨、妖股。
    
    4. 分歧一致引擎
    识别点火、发酵、加速、分歧、再一致、退潮。
    
    5. 风险兑现引擎
    识别高位兑现、亏钱效应、主线衰退、周期切换。
    
    6. 历史复盘引擎
    用历史相似周期验证当前判断，不让系统只靠主观打分。
    ```
    
    **一句话定稿**
    
    咱们这个系统的灵魂是：
    
    **用周期判断“时”，用主线判断“机”，用龙头判断“势”，用分歧一致判断“点”，用风险模型判断“退”。**
    
    后续所有页面、接口、数据库和算法，都围绕这句话展开。

    一期可以做，但要明确分级：
    
    ```
    历史行情复原 = 可以做
    历史板块成分精确复原 = 目前做不到完全准确
    ```

    ```
    真实历史事实
    推断历史事实
    当前关系回填
    人工确认结果
    ```
    
    不能把“用当前板块关系回推历史”伪装成真实历史。
    
    **一期处理方案**
    
    先采用三层策略：
    
    ```
    1. 股票自身事实：准确
    股票日K、涨跌幅、成交额、连板、高度、炸板等，基于个股行情可以准确复原。
    
    2. 板块行情事实：较准确
    板块日K、板块涨跌、成交、热度，如果有板块历史行情，就可以准确复原板块强弱。

    如果只有当前关系，就用当前关系回填历史，但标记为 INFERRED_CURRENT_RELATION。
    ```

    ```

    全市场涨跌
    成交额
    涨停跌停
    赚钱亏钱效应
    ```
    
    主线判断会受影响，因为主线需要：

    ```
    板块内涨停家数
    板块内梯队
    龙头带动板块
    个股所属概念
    ```
    
    龙头判断中，个股高度和换手可以准确，但“它属于哪个板块、带动哪个板块”会有误差。

    后续数据库需要设计成版本化关系：

    ```
    stock_plate_relation_snapshot
    - trade_date
    - stock_code
    - plate_code
    - relation_type
    - source
    - confidence
    ```
    
    一期先这样做：
    
    ```

    trade_date = 历史日期
    relation_source = CURRENT_BACKFILL
    confidence = 0.6
    ```
    
    以后如果爬到了历史成分，就覆盖或新增：

    ```
    relation_source = HISTORICAL_CRAWLED
    confidence = 0.95
    ```
    
    人工修正后：
    
    ```
    relation_source = MANUAL_CONFIRMED
    confidence = 1.0
    ```
    
    **核心原则**
    
    你现在的数据足够先把一期跑起来，但系统必须诚实：
    
    ```
    能确定的，标记为事实
    推断出来的，标记为推断
    人工确认的，标记为确认
    ```
    
    这样就不会污染训练样本。

    ```
    中可信样本：股票事实 + 当前关系回推
    低可信样本：关键字段缺失
    ```

- 一期页面与接口VO 设计细节
    
    下面是 **《一期页面与接口 VO 设计终稿》**。
    
    目标：把《顿悟股道》的思想，通过页面、卡片、接口、VO 完整表达出来。所有页面都支持历史任意交易日，所有结论都必须有证据链、冲突证据、置信度、规则版本、数据质量、明日验证点。
    
    ---
    
    # 一、全局页面规则
    
    ## 1. 所有页面统一支持日期选择
    
    所有接口必须支持：
    
    ```
    tradeDate       被查看的交易日
    asOfDate        判断站点日期，不传默认等于tradeDate
    judgementMode   REALTIME / RETROSPECTIVE
    ```
    
    含义：
    
    ```
    REALTIME：只使用 tradeDate 当天及以前数据，模拟站在当日收盘后判断
    RETROSPECTIVE：允许结合后验验证结果，用于复盘和规则校准
    ```
    
    ## 2. 所有判断卡片必须展示
    
    ```
    结论
    置信度
    规则版本
    数据质量
    支持证据 evidenceList
    冲突证据 conflictList
    风险提示 warningList
    明日/未来观察点 nextWatchList
    ```
    
    ## 3. 通用接口参数
    
    ```
    public class MarketQuery {
        private LocalDate tradeDate;
        private LocalDate asOfDate;
        private String judgementMode; // REALTIME / RETROSPECTIVE
    }
    ```

    # 五、规则执行机制
    
    ## 1. 单因子执行
    
    流程：
    
    ```
    读取 factor_definition
    读取 rule_factor_weight
    计算 FactorResult
    转成 evidence/conflict/warning
    参与 confidence
    ```
    
    ## 2. 组合因子执行
    
    流程：
    
    ```
    读取 factor_combination_definition
    判断组合是否触发
    写 factor_combination_trigger_record
    影响 confidence / risk / conclusion
    后续验证组合因子表现
    ```
    
    ## 3. 置信度计算
    
    基础公式：
    
    ```
    confidence =
    支持证据得分
    - 冲突证据惩罚
    - 数据质量降级
    - 组合风险降级
    ```
    
    要求：
    
    ```
    不能输出100%确定
    有冲突证据必须降低 confidence
    数据不足必须降低 confidence
    ```

- 后端-前端架构你想怎么设计
    
    我的想法是：**后端按“推演引擎系统”设计，前端按“复盘工作台系统”设计。**
    
    不是普通前后端 CRUD，也不是行情看板。
    
    **后端架构**
    
    后端我会分成三条主线：
    
    ```
    1. 数据线：爬取、清洗、审计、快照
    2. 推演线：周期、主线、龙头、分歧、风险
    3. 成长线：验证、修正、经验、规则权重
    ```
    
    模块上：
    
    ```
    module-spider
    负责采集股票、板块、指数、涨跌停、资金流、板块关系。
    
    module-market-data
    负责封装 ClickHouse 事实数据查询，不让引擎直接到处查表。
    
    module-engine
    核心推演模块，放 Cycle/Mainline/Leader/Divergence/Risk 等引擎。
    
    module-validation
    负责 T+1/T+3/T+5/T+10 验证。
    
    module-experience
    负责因子表现、组合因子表现、系统成长日志。
    
    module-review
    负责人工修正、证据标注、历史样本确认。
    
    module-api
    负责前端接口和 VO 组装。
    ```
    
    代码核心抽象必须统一：
    
    ```
    JudgementResult<T>
    EvidenceItem
    RuleContext
    FactorResult
    DataQualityContext
    NextWatchItem
    ValidationResult
    ```
    
    所有引擎都必须返回 `JudgementResult<T>`，不能只返回一个标签。
    
    我会强制引擎输出：
    
    ```

    confidence
    ruleVersion
    dataQualityLevel
    evidenceList
    conflictList
    warningList
    nextWatchList
    ```

    **我的最终设计取向**
    
    后端像一个有纪律的推演引擎：
    
