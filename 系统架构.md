二、引擎编排 — 已经编排了
编排器: EngineBatchOrchestrator
引擎跑批是 3 步串行流水线，每步有依赖检查：
顺序	Task	做什么	依赖
10	DataQualityCheckTask	检查 3 个核心事实表是否有数据（market_breadth / stock_limit_status / plate_daily），无数据则阻断当日推演	无
20	DailyJudgementTask	调用 DailyInferencePipeline.infer() → 所有引擎（CYCLE/MAINLINE/LEADER/DIVERGENCE/RISK/MARKET_OVERVIEW）并行推理 → 证据入库 + 前哨项入库	DATA_QUALITY_CHECK 成功
30	ForwardValidationExperienceTask	回测前一天的前哨项是否命中 → 更新经验因子	ENGINE_JUDGEMENT 成功

调用入口: POST /api/batch/daily?tradeDate=2026-07-08

五、历史数据跑批方案
MySQL 表已有: engine_batch_run_log（跑批日志）— 记录了每次跑批的 batchId / tradeDate / status / 耗时
断点续传: 爬虫有 spider_task_checkpoint（taskCode + tradeDate + provider → 已成功则跳过）
历史跑批步骤：
先跑 Spider（补历史事实表数据）：
# 东财单日
curl "http://localhost:8080/api/spider/dc/daily?tradeDate=2026-07-01"
# 同花顺单日
curl "http://localhost:8080/api/spider/ths/daily?tradeDate=2026-07-01"
需要逐日调用。Spider 有断点续传，重跑同一日会跳过已成功任务。

再跑引擎（从快照表读事实数据，写入快照）：
curl -X POST "http://localhost:8080/api/batch/daily?tradeDate=2026-07-01"
引擎跑批也有日志表，但不自动幂等（重跑会重复写入快照，ClickHouse ReplacingMergeTree 自动去重）。

前验任务（可选，需要至少 2 天数据）：
# 在 asOfDate 晚于 tradeDate 时自动触发
curl -X POST "http://localhost:8080/api/batch/daily?tradeDate=2026-07-01&asOfDate=2026-07-02"

六、当前缺口（不做的就跑不起来）
缺口	严重度	说明
plate_daily_snapshot 聚合逻辑	🔴 P0	DataQualityCheck 检查此表，没数据就阻断引擎
market_breadth_daily_snapshot 聚合逻辑	🔴 P0	同上
每日 SQL 建表	🔴 P0	ClickHouse + MySQL 表需先建
trading_calendar 填充	🟡 P1	所有引擎依赖日历
stock_pool_daily_snapshot 缺 source + type 字段	🟡 P1	用户之前要求加的
历史自动化（循环多日跑批脚本）	🟡 P2	需要写个 shell 脚本逐日调

关键结论：plate_daily_snapshot 和 market_breadth_daily_snapshot 的聚合计算逻辑必须补上，否则 DataQualityCheck 第一步就失败，后续引擎全部不会跑，前端所有菜单都没数据。
